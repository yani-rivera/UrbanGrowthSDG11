# orchestrator_config.json

## Purpose

The `orchestrator_config.json` file is the master configuration file controlling execution of the SDG-11 Real Estate Framework.

Rather than hard-coding paths, processing steps, aggregation outputs, and auxiliary resources inside the software, these settings are externalized into a single configuration file. This design improves reproducibility, maintainability, and portability.

The orchestrator reads this file at runtime to determine:

* Which processing steps are available
* The order in which steps should execute
* The location of scripts and resources
* Aggregation products to generate
* Logging behavior
* Reporting options

---

## Structure Overview

```text
orchestrator
├── logging
├── aggregation_tasks
├── paths
├── configs
├── catalogs
├── scripts
├── pipeline
├── reporting
└── report_metrics
```

## orchestrator

Stores basic metadata regarding the orchestrator version.

### Example

```json
{
  "orchestrator": {
    "version": "SDG11_ORCHESTRATOR_V3"
  }
}
```

## logging

Defines the fields recorded in execution logs.

These logs support:

* Auditability
* Error tracking
* Quality control
* Workflow reproducibility

### Recorded Fields

| Field                | Description                 |
| -------------------- | --------------------------- |
| run_id               | Unique execution identifier |
| run_date             | Execution timestamp         |
| step                 | Pipeline step executed      |
| agency               | Agency name                 |
| agency_folder        | Agency directory            |
| mnemonic             | Agency mnemonic             |
| year                 | Processing year             |
| month                | Processing month            |
| input_file           | Source file                 |
| input_path           | Source path                 |
| config_file          | Configuration used          |
| output_file_expected | Expected output             |
| status               | Success or failure          |
| return_code          | Script return code          |
| error_message        | Error description           |
| metrics_json         | Execution metrics           |

## aggregation_tasks

Defines all aggregation products generated by the framework.

Each task specifies:

* Aggregation name
* Script to execute
* Output filename
* Whether yearly parameters are required

Current aggregation products include:

| Task         | Purpose                         |
| ------------ | ------------------------------- |
| neighborhood | Neighborhood-level summaries    |
| bedrooms     | Bedroom-based summaries         |
| area         | Area-based summaries            |
| area_beds    | Combined area-bedroom summaries |

New aggregation products can be added without modifying orchestrator code.

## pipeline

Defines the default workflow sequence executed by the orchestrator.

### Current Sequence

```text
parse
→ merge
→ deduplicate
→ filter
→ uid
→ clean
→ stdprice
→ transaction_validate
→ aggregate
```

Individual steps may be executed independently or combined into a full end-to-end workflow.

## reporting

Controls report generation.

### Options

| Parameter              | Purpose                         |
| ---------------------- | ------------------------------- |
| enabled                | Enables reporting               |
| save_step_reports      | Saves reports for each stage    |
| save_execution_summary | Saves overall execution summary |

---

## report_metrics

Stores quality-control metrics generated by validation stages.

Typical metrics include:

* Successfully validated records
* Corrected transaction classifications
* Removed invalid prices
* Out-of-range detections

These metrics support workflow monitoring and quality assurance.

---

## Design Philosophy

The orchestrator configuration file serves as the central control layer of the framework. By externalizing workflow definitions, paths, scripts, catalogs, and processing parameters, the system remains reproducible, configurable, and easier to maintain. Most operational changes can be implemented through configuration updates without modifying the orchestrator source code itself.
